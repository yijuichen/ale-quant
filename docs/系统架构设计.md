**文档目标:** 定义 QuantSaaS 全局有哪些物理端、有哪些逻辑模块、状态如何在它们之间流转，以及系统整体的生命周期动作。不含任何具体策略公式。

---

## 1. 架构哲学：时空正交与无记忆解算

**动作原则：** 所有业务推进必须被封装为离散的"动作"，动作之间严禁隐式状态传递（禁止"欠债记录"）。每个动作读入当前状态快照，产出新的确定性意图，不依赖上一动作的内存变量。

**窗口原则：** 每个模块只能读取属于自己的"数据窗口"和"资金窗口"。策略模块看不到数据库内部状态，宏观引擎看不到微观的浮仓细节，进化引擎看不到实盘账户。

## 2. 三端物理部署形态

### 2.1 SaaS 管理端（云端，`app_role: saas`）

对策略模板、实例、用户认证与订阅进行全量管理。通过 cron 扫描 RUNNING 实例，并仅在该实例最新已完成的 `[聚合周期]` 聚合桶尚未处理时执行一次 `Step()`。通过 WebSocket Hub 下发交易指令给连接中的 LocalAgent，并接收成交上报。

**不执行：** GA 进化与回测运算（可通过 `app_role` 在路由层拦截写接口）。

**不保存：** 任何交易所 API 凭证。


### 2.2 LocalAgent 执行端（用户本地）

极简二进制进程，用户本地运行。配置文件（`config.agent.yaml`）内保存 SaaS 连接信息与 [你的交易所] API 凭证。

职责：
1. 启动时通过 SaaS REST 登录获取 JWT
2. 用 JWT 建立 WebSocket 长连接到 SaaS `/ws/agent`
3. 收到 `TradeCommand` 后调用 [你的交易所] REST API 执行现货下单
4. 获取成交明细与当前余额，构建 `DeltaReport` 上报 SaaS
5. 定时发送心跳；断线后以指数退避自动重连

**不含：** 任何策略代码，不连接数据库，不自主决策。


### 2.3 进化算力面（本地实验室，`app_role: lab`）

也是完整 SaaS 系统，但部署在本地算力机器上。数据库连接云端 SaaS 的同一 Postgres 实例，专注执行 GA 进化任务与回测运算。

**不执行：** 真实交易指令下发（`app_role: lab` 在路由层禁用实例启停与交易下发）。


### 2.4 `app_role` 三态行为矩阵

| 角色 | 部署场景 | 开放能力 | 限制 |
|------|---------|---------|------|
| `saas` | 云端 SaaS | 实例管理、交易指令下发、用户运营 | 禁止进化/回测写接口 |
| `lab` | 本地实验室 | GA 进化、回测、基因库管理 | 禁止交易指令下发、实例只读 |
| `dev` | 开发测试 | 所有模块全开 | 无限制 |

## 3. 逻辑模块与责任边界

### 3.1 Strategy 策略模块（产品中心）

管理策略模板（图纸）。定义策略的参数结构、版本标识、交易对类型（现货/合约）。当前主线策略为 `[your-strategy-id]`，面向 [你的交易所] 现货 [你的交易对]。

`Step()` 是策略决策的唯一外部入口，所有买卖意图、仓位规模、风控判断均在 `Step()` 内产生。回测路径与实盘路径共用同一套 `Step()` 实现，不允许内部分叉。


### 3.2 Live/Instance 交易执行模块（设备中心）

实例（Instance）的本质是：策略模板 + 交易标的 + 资金配额 + 用户授权。实例的创建、启动、停止、删除，以及运行中的状态（`RUNNING` / `STOPPED` / `ERROR`）全部由此模块管理。

cron 按基础扫描频率检查所有 `RUNNING` 状态实例；实例 tick 只有在最新已完成的 `[聚合周期]` 聚合桶未处理过时才推进：拉取公开 K 线数据，读取 `PortfolioState`，构建 `StrategyInput`，调用 `Step()`，将产出的交易意图翻译为 `TradeCommand`，通过 WebSocket Hub 下发给已连接的对应 Agent。


### 3.3 Evolution & Backtest 模块（参数实验室）

通过 GA 遗传算法对策略参数进行进化迭代，产出 **挑战者（challenger）** 参数包。人工在前端审批后可将挑战者晋升为 **冠军（champion）**。冠军参数包用于实例创建或参数更新。

只在 `app_role: lab` 或 `dev` 下开放进化任务创建与执行，回测路径调用与实盘相同的策略 `Step()` 实现。


### 3.4 User/Auth 模块

处理用户注册、登录、JWT 签发，以及订阅计划校验（创建实例、启动实例时做配额守门）。此模块不涉及策略逻辑，不在本文档细化。

## 4. 全局状态总线

### 4.1 单一 Postgres + 单一 Redis

整个系统使用单一 Postgres 数据库（`quantsaas`），不再区分 Shell DB 与 Live DB。Redis 仅作缓存使用（冠军基因 key、会话等），不承担信号传递职责。


### 4.2 数据真源与所有权

| 数据 | 真源 | 说明 |
|------|------|------|
| 策略决策逻辑 | `Step()` 实现（SaaS 侧执行） | 回测与实盘必须共用 |
| 冠军参数包 / 模板 | SaaS DB | 进化产出，供实例绑定 |
| 终端用户与订阅 | SaaS DB | 统一在 SaaS 管理 |
| 运行中实例状态 | SaaS DB | 由 Agent 上报后维护 |
| 交易所 API 凭证 | LocalAgent 本地配置文件 | 仅 Agent 持有，永不进入 SaaS DB |
| 成交记录与资产快照 | SaaS DB | Agent 上报后持久化 |
| GA 任务与基因记录 | SaaS DB | 单一数据库，无分库 |


### 4.3 交易所凭证物理隔离铁律

[你的交易所] API Key / Secret / Passphrase **只能**存在于 LocalAgent 的本地配置文件（`config.agent.yaml`）中。

- 永不进入 SaaS 服务
- 永不写入任何数据库表
- 永不通过网络传输到云端

发现代码将 API Key 写入 SaaS 侧时，必须立即停止并报告。

## 5. IoT 下发与上报通信协议

云端与端侧严格遵循"**云端只信上报，端侧无脑执行**"机制，彻底解决网络断线、丢包与脏数据问题。

### 5.1 WebSocket 长连接模型

Agent 发起连接，SaaS 监听 `/ws/agent` 端点。连接建立时 Agent 必须先发送含 JWT 的鉴权消息，SaaS 验证通过后返回鉴权结果，之后进入正常收发循环。Agent 每 30 秒发送一次心跳，SaaS 回确认。


### 5.2 消息类型全表

| 消息类型 | 方向 | 触发时机 |
|---------|------|---------|
| `auth` | Agent → SaaS | 连接建立后立即发送，携带 JWT |
| `auth_result` | SaaS → Agent | 鉴权完成后返回 |
| `heartbeat` | Agent → SaaS | 每 30 秒 |
| `heartbeat_ack` | SaaS → Agent | 收到心跳后 |
| `command` | SaaS → Agent | `Step()` 产出意图后，SaaS 翻译为具体指令下发 |
| `command_ack` | Agent → SaaS | 收到 command 后立即确认（不等执行完成） |
| `delta_report` | Agent → SaaS | 交易执行完成后；或重连时的初始余额快照 |
| `report_ack` | SaaS → Agent | SaaS 成功处理 DeltaReport 后 |


### 5.3 TradeCommand 语义

每条 `TradeCommand` 携带以下语义字段：

| 字段 | 取值范围 | 说明 |
|------|---------|------|
| `client_order_id` | 全局唯一字符串 | 格式：`inst{id}-{type}-{ts}`，用于去重与审计 |
| `action` | `BUY` / `SELL`（大写） | 交易方向 |
| `engine` | `MACRO` / `MICRO`（大写） | 产生此指令的引擎层 |
| `symbol` | 如 `[你的交易对]` | 交易对 |
| `amount_usdt` | 字符串金额 | 买入时使用，表示花费 USDT 数量 |
| `qty_asset` | 字符串数量 | 卖出时使用，表示卖出标的资产数量 |
| `lot_type` | `DEAD_STACK` / `FLOATING` | 仓位语义，宏观吸入底仓 vs 微观浮动仓 |

SaaS 下发 TradeCommand 的同时在数据库写一条 pending 记录，供 DeltaReport 回填时按 `LotType` 拆分更新 `DeadStack` / `FloatStack`。


### 5.4 DeltaReport 语义

Agent 执行完毕后上报，包含：
- `balances`：当前所有资产的余额快照（标的资产与 USDT 的可用与冻结）
- `execution`：本次成交明细（成交量、成交价格、手续费、状态）
- `client_order_id`：对应的指令 ID（重连时发送初始快照时可为空）


### 5.5 状态收敛与天然自愈

SaaS 收到 DeltaReport 后刷新对应实例的 `PortfolioState`，持久化 `SpotExecution` 与 `TradeRecord`，并写审计日志。

下次 `Step()` 调用时，系统基于真实余额与语义拆分重新决策。这意味着：
- Agent 断线后 SaaS 跳过该实例的 tick，无须人工干预
- Agent 重连后立即发送初始余额快照，SaaS 在下次 tick 自动纠偏
- SaaS 重启后从 DB 恢复 RUNNING 实例列表，等待 Agent 重连

## 6. 系统级生命周期动作

### 6.1 系统初始化（System Init）

**触发时机：** SaaS 进程启动时。

**动作流转：**
1. 读取配置文件（`config.yaml`），初始化 DB 连接、Redis 连接、JWT 秘钥
2. 执行 GORM AutoMigrate，以 Go struct 为唯一 schema 真源完成数据库结构同步
3. 从 DB 读取所有状态为 `RUNNING` 的实例，注册到内存管理列表
4. 启动 WebSocket Hub，开始接受 Agent 连接
5. 启动 cron 调度器，注册实例基础扫描任务；具体 `Step()` 推进由每个实例的聚合桶去重控制


### 6.2 Cron Tick 驱动实例 Step()

**触发时机：** cron 基础扫描命中 `RUNNING` 实例后，若该实例最新已完成的聚合桶尚未处理，则触发一次 `Step()`；已处理过同一桶时跳过，避免实盘比回测多跑决策。

**动作流转：**
1. 从交易所公开 API 拉取 K 线历史数据
2. 从 DB 读取实例的 `PortfolioState`（由上次 DeltaReport 维护的真实账户状态）
3. 从 DB 读取 `RuntimeState`（策略内部运行状态快照）
4. 构建 `StrategyInput`（含 K 线序列、资产状态、时间戳）
5. 调用 `panel.Step()`（**只在 SaaS 侧执行**，Agent 不含此逻辑）
6. 将 `Step()` 产出的交易意图翻译为 `TradeCommand`（含全局唯一 `client_order_id`）
7. 在 DB 写入 pending 执行记录
8. 通过 WebSocket Hub 将 `TradeCommand` 下发给对应 Agent

若 Agent 当前未连接，则跳过下发，记录警告，等待下次 tick 时重试。


### 6.3 Agent 断线重连

**触发时机：** Agent 检测到 WebSocket 连接断开。

**动作流转：**
1. 等待初始退避时间（1 秒），尝试重新登录获取 JWT
2. 建立新的 WebSocket 连接
3. 连接建立后立即发送初始 `DeltaReport`（当前交易所余额快照，不含 `client_order_id` 与 `execution`）
4. SaaS 收到快照后更新相关实例的 `PortfolioState`
5. 下次 cron tick 时，SaaS 基于真实余额重新决策，无需人工干预

重连退避策略：初始 1 秒，每次翻倍，最大等待 5 分钟。


### 6.4 优雅停机与状态快照（Graceful Shutdown）

**触发时机：** SaaS 进程接收到操作系统中断信号（如 `SIGTERM`）。

**动作流转：**
1. 停止接受新的 Agent 连接与 cron tick 任务
2. 等待正在执行中的 tick 完成（超时保护）
3. 持久化所有活跃实例的 `RuntimeState` 快照，避免重启后策略状态撕裂
4. 关闭所有 WebSocket 连接，发送关闭帧
5. 关闭 DB 与 Redis 连接

## 7. 不可推翻的技术决策

以下为整个系统的硬约束，违反时必须停止开发并报告，不得绕过。

**1. 复利前置条件（铁律）**
任何涉及策略机制的设计或实现，必须能清楚说明复利如何发生（资金规模随权益正反馈滚动）。不满足可论证复利性质的策略设计，不进行开发。


**2. 策略同构原则**
回测与实盘必须调用同一个 `Step()` 实现。`Step()` 内部禁止出现 `if isBacktest { ... } else { ... }` 类分支。`Step()` 只在 SaaS 侧执行，Agent 二进制不包含任何策略代码。


**3. 策略内部纯净原则**
策略代码内部绝对禁止：定时器、网络请求（HTTP / WebSocket / gRPC）、数据库读写、任何文件 I/O。策略是纯函数，只依赖传入的 `StrategyInput`，只输出 `StrategyOutput`。


**4. 现货策略 ACL OHLC 剥离**
`Manifest.IsSpot == true` 的策略，内核代码禁止依赖 `OHLCV` 聚合结构（`quant.Bar`）。所有 OHLCV 到 `[]float64 + []int64`（收盘价序列 + 时间戳序列）的降级必须在 ACL 外圈（回测适配器入口与实盘 tick 入口）完成。


**5. API Key 物理隔离**
[你的交易所] API Key / Secret / Passphrase 只允许存在于 LocalAgent 的本地配置文件中，永不进入 SaaS，永不写入数据库，永不通过网络传输。


**6. GORM AutoMigrate 为唯一 schema 真源**
数据库结构以 Go struct 为真源，通过 GORM AutoMigrate 管理。不写 SQL migration 文件，不维护版本化 migration 脚本。


**7. 无量纲计算**
所有价格相关计算使用对数收益率或比率（无量纲），禁止用绝对价格做跨标的比较。


**8. 仓位三态**
资产状态以 `DeadStack`（宏观长期底仓）/ `FloatStack`（微观浮动仓）/ `ColdSealedStack`（冷封存，永不释放）三态区分语义，不能混用。仓位三态的更新规则由策略 `Step()` 决定，SaaS 按 `LotType` 字段落账。


**9. 底仓释放仅在 SaaS 侧更新，不下发 Agent**
`dead_stack → floating` 的语义转换只改变 SaaS 侧的 `PortfolioState` 账本，不向 Agent 下发任何 `TradeCommand`，但必须写审计日志。


**10. 单一数据库，无分库**
SaaS 使用单一 Postgres 实例，不分 Shell DB 与 Live DB，不使用 Redis 作为信号传递通道（Redis 仅作缓存）。


**11. 用户界面零数学**
面向用户的字符串中避免出现内部状态机术语、无上下文的裸数学量名或公式片段。希腊字母并非一律禁止；若含义上必须保留，应采用「字母 + 文字释义」并置（例如「σ（标准差）」「ρ（相关系数）」），禁止仅展示希腊字母而无对应说明。
