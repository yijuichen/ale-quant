**文档目标:** 将进化系统降维为一个纯粹的寻优黑盒。进化引擎不关心具体策略内部结构，只通过抽象接口驱动种群生命周期。具体策略的染色体字段、边界、适应度语义由策略模块在各自文档中定义。

---

## 0. 核心定位

进化引擎是一个**纯计算线程**：接收策略提供的基因空间与评估函数，对种群执行多代进化，交付最优基因集（Champion）。

引擎不关心：策略内部的信号公式、仓位三态语义、特定时钟机制，也不 import 任何具体策略实现。引擎只知道：基因是一个可采样、可变异、可交叉、可评分的抽象向量。

**输入：** 策略提供的基因空间（边界、约束）+ 历史 K 线数据集 + 进化配置参数

**输出：** 挑战者（challenger）基因包 JSON，写入数据库，等待人工 Promote 为冠军

## 1. 探索窗口定义

### 1.1 数据切片窗口（多时段坩埚）

Epoch 启动时，引擎一次性从数据库拉取**本 Epoch 所需的全部**历史 K 线序列用于构造坩埚：**不对全库长度设「最多十年」等人工上限**；能有多长取决于上游已同步入 Postgres 的数据，而下限以**交易所侧可拉取到的最早可用 bar** 为准（例如主流现货对能同步多早就多早，直至 API/上架历史起点）。在此基础上再构造以下四个滚动窗口（标识「10y」表示长周期档，**评分区间取入库全序列**——自最早可用 bar 至最新 bar，不设人工天数上限）。

| 窗口标识 | 目标时长（评分截取区间） | 适应度权重 |
|---------|---------|---------|
| 10y | 全量（最早至最新 bar） | 0.40 |
| 5y | 1825 天 | 0.30 |
| 2y | 730 天 | 0.20 |
| 6m | 183 天 | 0.10 |

表中「目标时长」：5y / 2y / 6m 为对齐最新一根 bar 的最近若干日历日，并在该截取区间之前再预留最多 1200 天 warmup K 线（用于指标预热，不计入评分区间）。**10y 档不再单独截断为固定天数**，直接使用本 Epoch 传入的 `bars` 全序列；若序列长于十年，则长周期评分自然覆盖更长历史。更短窗口仍使用同一全库序列中的尾部片段及各自 warm-up 前缀。

**严禁未来数据泄露：** 每个窗口的评估区间从 `EvalStartMs` 开始，warmup 数据必须在 `EvalStartMs` 之前。


### 1.2 基因空间窗口

策略通过以下语义向引擎提供基因边界，引擎不直接读取字段名：

- **采样（Sample）：** 从合法空间随机生成一个基因实例
- **修复（Clamp）：** 将越界基因修复到合法范围（含结构约束修复）
- **验证（Validate）：** 检查基因是否合法


### 1.3 出生点冻结窗口（Epoch 级）

**`quant.SpawnPoint`**（`CapitalPolicy` + `RiskBounds`）在 Epoch 启动时固定，全种群共享；不参与代内交叉变异，不进入基因组指纹。

当前 QuantSaaS 实现：`POST /api/v1/evolution/tasks` 支持 `spawn_mode`：`inherit`（冠军包或默认）、`random_once`（排队时抽样一次并冻结）、`manual`（请求体填写 `spawn_point`）。引擎经 `EpochConfig.SpawnPointOverride` 注入 `internal/saas/ga`（见 `handler_evolution.go`、`engine.go`）。

## 2. 进化生命周期动作

### 2.1 种群初始化

种群大小默认 `Population = 300`，初始化策略：

当数据库中存在精英基因时：
- **10%**：从数据库精英中原样拷贝
- **40%**：在精英基础上施加加性高斯强化变异（初始化路径固定变异概率 `0.15`、尺度 `1.5`，与代内 `MutationRamp` 无关）
- **50%**：完全随机生成

当数据库无精英时：第一个体为产品默认冠军染色体原样，其余个体完全随机生成。

实现约定：索引 0 始终为当前种子冠军原样（有 DB 精英则为库中冠军染色体，否则为产品默认冠军）；仅对索引 1 起的剩余个体按上表约 10% / 40% / 50% 计数分配（相对剩余个体数四舍五入）；无 DB 精英时索引 1 起全部为 `RandomChromosome`。


### 2.2 并发适应度评估

评估采用固定大小 worker pool 并发执行：

$$Workers = \min(NumCPU,\; PopulationSize)$$

每个 worker 独占一份 Adapter 实例（回测上下文），无共享可变状态。所有 worker 并发处理任务队列，等所有任务完成后进入下一代。


### 2.3 选择（锦标赛 Tournament）

每次选择从种群中随机抽 `TournamentSize`（默认 3）个个体，取适应度最高者为父代。适应度为 `fatalFitnessScore`（灾难性极小值）的个体在锦标赛中自然被淘汰（无需显式排除，因 fatal 个体分数极低）。


### 2.4 交叉（Uniform Crossover）

对每个基因维度以 0.5 概率独立地从两个父代中选取，生成子代。引擎不关心字段语义，由策略的 `Clamp` 逻辑在交叉后修复结构约束。

当前实现：`uniformCrossoverWithMode` 对所有维度独立 50% 交换，结构约束修复在交叉后调用 `repairGeneWithMode`。宏观/微观块正交交叉为[规划，未实现]。


### 2.5 变异（加性高斯）

每个基因维度以独立 Bernoulli 概率 `MutationProbability`（默认 0.15）决定是否变异；变异量为 `NormFloat64() × GeneStep × MutationScale`（默认 Scale=1.0）。变异后调用 Clamp 修复越界值。


### 2.6 精英保留（Elitism）

每代保留适应度 Top `Elitism`（默认 8）个个体，不经交叉变异直接入下一代。


### 2.7 收敛检测与变异斜坡（Mutation Ramp）

连续 `EarlyStopPatience`（默认 5）代内最佳适应度改善低于 `EarlyStopMinDelta`（默认 0.001）时，触发变异斜坡：

$$MutationProbability \mathrel{\times}= MutationRampProbFactor \;(1.25)$$

$$MutationScale \mathrel{\times}= MutationRampScaleFactor \;(1.25)$$

上限：`MutationProbabilityMax = 0.55`，`MutationScaleMax = 3.0`

当变异参数触及上限后仍无改善，才真正提前终止（Early Stop）。


### 2.8 Genome 指纹缓存（Fingerprint Cache）

策略提供 Fingerprint（指纹）语义（对基因向量按精度 1e-6 哈希），引擎在 Epoch 内检测重复个体，命中缓存则复用评估结果，跳过重复回测。预期命中率约 15–30%（来自精英保留产生的重复）。

### 2.9 收敛交付

进化结束后（正常完成或 Early Stop），将最优基因包编码为 JSON（`EncodeResult`），写入数据库为 `challenger` 角色。进化任务不自动 Promote，必须等待人工审批。

## 3. 适应度黑盒（Fitness Function）

### 3.1 多窗口坩埚分数

对每个切片窗口，以策略的累计回报相对于 Ghost DCA 基准计算 Alpha，再按超额回撤惩罚：

$$Alpha = ROI_{strategy} - ROI_{GhostDCA}$$

$$SliceScore = Alpha - 1.5 \times \max(0,\; MaxDD_{strategy} - MaxDD_{GhostDCA})$$

当 $MaxDD_{strategy} \geq 0.88$ 时，`SliceScore = -99999`（硬否决，fatal）。

各窗口加权汇总：

$$ScoreTotal = 0.40 \times Score_{10y} + 0.30 \times Score_{5y} + 0.20 \times Score_{2y} + 0.10 \times Score_{6m}$$


### 3.2 Ghost DCA 基准

`SimulateGhostDCA` 以种子资本买入标的资产，然后每自然月月初注入月度预算买入所有可用 USDT，作为被动 DCA 基准。策略个体必须跑赢此基准（Alpha > 0）才算有效。

注资节奏：月度 `monthlyInject` USDT，在每个自然月边界触发。


### 3.3 ROI 计算

策略的 ROI 采用"Modified Dietz 收益率"思路，剔除注资跳变对 NAV 的影响（注资时刻的权益变化不计入收益）。NAV 最大回撤基于净资产曲线计算。


### 3.4 级联短路（Cascading Short-circuit）

适应度评估按切片窗口从短到长（6m → 2y → 5y → 10y）顺序执行。一旦某个窗口触发 fatal（MaxDD ≥ 88%），立即返回 fatal 分数，跳过后续更长的窗口。

## 4. 性能优化机制

以下优化已在当前实现中落地：

**4.1 级联短路：** 按窗口长度升序评估，fatal 立即终止（见 3.4）。

**4.2 并发评估：** Genome 粒度 worker pool，CPU 数量上限（见 2.2）。

**4.3 Tournament fatal 排除：** fatal 个体因分数极低，在锦标赛中几乎不会被选中（无需显式过滤）。

以下优化属于**规划，未实现**：

**4.4 Nested Replay（嵌套重放）：** 一次全样本回测 + 各窗口 EvalStart checkpoint 差分，将 4 次独立回测合并为 1 次，减少 75% 计算量。[规划，未实现] **注意：此优化与 4.1 级联短路存在根本冲突**——Nested Replay 必须先跑完整 10y 序列才能提取子窗口结果，无法在 6m fatal 时提前退出；而早期代 fatal 率通常 40–60%，10y bar 量约为 6m 的 10 倍，实施后反而劣化约 4×。仅在 fatal 率极低（< 10%）且短路收益可忽略时才有净收益，当前不宜实现。

**4.5 前缀和 O(1) 滑窗统计 + EMA 增量化：** 减少每次适应度评估中的重复统计计算。[规划，未实现]

**4.6 蒙特卡洛异步化：** Epoch 验证阶段的 MC 采样并行化，主线程等待结果。[规划，未实现]

## 5. Epoch 验证与蒙特卡洛

Epoch 进化结束后，对产出的冠军候选执行额外验证（属于[规划，未实现]，当前实现跳过此步骤）：

**5.1 全样本复核：** 对冠军在全样本 1m 精度数据上重跑一次，确保 Adapter 内嵌的指标预热不影响结果。

**5.2 蒙特卡洛风险评估：** 对冠军的成交对数收益序列执行 `RunMonteCarlo`，输出：
- 破产概率（Ruin Probability）
- 收益分位数（第 5、50、95 百分位的终局权益）

## 6. 结果交付与基因角色

### 6.1 基因角色三态

数据库中的基因记录有三种角色，形成状态机：

| 角色 | 含义 |
|------|------|
| `challenger` | 进化产出，等待人工审批 |
| `champion` | 当前活跃冠军，驱动实盘实例 |
| `retired` | 历史冠军，仅供查阅与回溯 |


### 6.2 人工 Promote 流程

进化任务产出 `challenger` 后，操作员在前端查看回测报告与风险指标，决定是否晋升。晋升操作在数据库事务内执行：

1. 当前 `champion` 改为 `retired`
2. 选定的 `challenger` 改为 `champion`，记录激活时间

晋升完成后，SaaS 的冠军缓存（Redis key）立即失效，下次 cron tick 时自动加载新冠军。


### 6.3 规划但未实现（当前 Epoch 不含）

以下机制**规划但未实现**，不应纳入当前开发范围：
- **WFO 推进式滚动测试（Walk-Forward Optimization）：** 按时间轴向前推进的滚动回测，防止过拟合
- **定时自动触发 GA：** cron 定期自动启动进化任务，无需人工触发

## 7. HTTP 触发契约（最小集合）

进化任务通过 REST API 触发，仅在 `app_role: lab` 或 `dev` 下可用。

### 7.1 创建进化任务

参数：
- `pop_size`：种群大小，默认 300，范围 [10, 500]
- `max_generations`：最大代数，默认 25，范围 [5, 50]
- `spawn_mode`（可选）：`inherit` | `random_once` | `manual`；`manual` 时须带 `spawn_point`（JSON，与 `quant.SpawnPoint` 对齐）
- `test_mode`（可选）：快速测试模式，Pop=10, Gen=3


### 7.2 查询任务状态与结果

返回当前进化任务状态（是否运行中）、历次 challenger 基因列表（含 ScoreTotal、MaxDrawdown、各窗口分数）。

## 8. EvolvableStrategy 抽象接口

进化引擎通过此接口与具体策略完全解耦，`engine.go` 不含任何策略特有的染色体字段名或类型引用。

### 8.1 八动词契约

| 动词 | 职责 |
|------|------|
| `StrategyID()` | 返回策略唯一标识符 |
| `Sample(rng)` | 从合法基因空间随机采样一个基因 |
| `Mutate(c, prob, scale, rng)` | 对基因施加加性高斯变异 |
| `Crossover(p1, p2, rng)` | 对两个父代执行交叉，产出子代 |
| `Fingerprint(c)` | 返回基因的唯一哈希（FNV-1a，精度 1e-6） |
| `Evaluate(ctx, c, plan)` | 在给定坩埚计划上评估基因，返回适应度与各窗口明细 |
| `DecodeElite(json)` | 从 DB ParamPack JSON 解码精英基因；`null` 时返回默认种子 |
| `EncodeResult(c, spawn)` | 将冠军基因与出生点序列化为 ParamPack JSON（`spawn_point` + `[strategy]_config`） |
| `Verify(ctx, c, spawn, bars, lotStep, lotMin)` | 验证路径全量回测（含 Monte Carlo），返回 `BacktestMetrics` |

`Gene = any`：引擎持有不透明载体，不读取内部字段。

### 8.2 EvaluablePlan 只读上下文

`EvaluablePlan` 在 Epoch 启动时由 `buildEvaluablePlan` 构建，在整个世代内不可变，传递给所有 `Evaluate` 调用：

```
EvaluablePlan
├── Pair, TemplateName           // 标的与模板标识
├── Spawn *quant.SpawnPoint      // 出生点（共享）
├── LotStep, LotMin              // 下单精度
├── Windows []CrucibleWindow     // 4 个切片窗口（含 K 线与权重）
├── DCABaselines []DCABaseline   // 预计算的 ghost-DCA 基线（与 Windows 一一对应）
└── AggregateCache               // 预聚合指标缓存（可选）
```

DCA 基线在 Epoch 启动时一次性计算，代内各染色体评估直接读取，不重复运行 DCA 回测。

### 8.3 引擎对染色体内部字段不可见原则

`engine.go` 通过八个动词操作 `Gene`，对染色体字段名完全不可见。新增策略（如 ETH、SOL 或其他自定义策略）只需实现 `EvolvableStrategy` 接口，无需修改引擎代码。

### 8.4 包位置约束（导入循环）

`[YourStrategy]Evolvable` 实现（含 `Run[YourStrategy]SingleBacktest`）位于 `internal/saas/ga/[strategy]_evolvable.go`，**而非** `internal/strategies/[strategy]/`。原因：

- `backtest → [strategy]`（`adapter.go` 使用策略的 `Params`）
- `genome → [strategy]`（`store.go` 使用 `[strategy].StrategyID`）

若将 `[YourStrategy]Evolvable` 放入策略包，则策略包须导入 `ga` 与 `backtest`，形成双重循环依赖。将其放在 `ga` 包内则无新循环（`ga` 已依赖策略包，为单向依赖）。
