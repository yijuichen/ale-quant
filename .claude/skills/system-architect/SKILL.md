# 系统架构师

## 何时使用

- 判断某功能该落在 `saas` / `agent` / `lab` 哪一端、`internal/` 哪一层
- 设计模块边界、状态流转、生命周期动作
- 审查改动是否违反三端物理隔离或执行边界

## 唯一真源

[docs/系统总体拓扑结构.md](../../../docs/系统总体拓扑结构.md)。文档未定义的架构不臆造。

## 不可推翻的边界

- **三端物理部署**：`saas`（决策大脑，执行 `Step()`，无 API Key）／`agent`（执行手，只下单上报，无策略代码）／`lab`（跑 GA 与回测，不下发真实交易）。
- **执行边界**：`Step()` 只在 SaaS 侧执行。
- **API Key 物理隔离**：凭证只存在于 `config.agent.yaml`，永不进入 SaaS。
- **时空正交、无记忆解算**：动作之间禁止隐式状态传递；每个动作读当前快照、产出确定性意图。
- **不做预防性解耦**：保持 SaaS-Strategy-Agent 分工，不为想象中的未来需求提前拆抽象。

## 工作方式

1. 先读 `docs/系统架构设计.md` 对应章节。
2. 把需求映射到具体端与目录（参见 [AGENTS.md](../../../AGENTS.md) 代码目录表）。
3. 任何跨端数据流，明确：哪端产出意图、哪端执行、凭证是否被隔离。
4. 发现需求超出文档定义范围时，停下并上报，不自行扩展。
